# Context

## 현재 상황 판단

지금 필요한 것은 "샘플 프로젝트를 Vue 3 + Spring Boot로 옮기는 솔루션"이 아니라, 운영 중인 대형 MES를 대상으로 한 AM 작업의 기본 틀이다.
다만 여기서 중요한 추가 전제는, 메뉴 / 결재 / 권한 / 인증 / 메일 같은 공통 기능은 이미 별도 공통 플랫폼이 제공하고 있다는 점이다.
따라서 이 저장소의 초점은 "독립 제품 개발"이 아니라 "공통 플랫폼 위에서 업무 페이지를 `Vue 3`로 전환하는 기준"이어야 한다.

핵심 제약은 아래와 같다.

- 실제 고객사 소스는 반출할 수 없다.
- 공개 샘플은 구조를 검증하기 위한 대체 입력셋일 뿐이다.
- 작업 대상은 단순 CRUD 예제가 아니라 대형 Smart Factory / MES 영역이다.
- 따라서 저장소의 주목표는 코드 생성기보다 분석 기준, canonical data model, 표준 산출물 체계를 만드는 것이다.
- 공통 플랫폼이 제공하는 횡단 기능은 본 저장소의 구현 범위 밖이다.

## 이 저장소의 역할

이 저장소는 다음 역할에 집중한다.

- 외부 지원 워크스페이스에서 사내 `AI Pro + GLM-4.7` 작업환경을 준비하는 하네스
- 레거시 자산을 분류하고 추적하는 공통 분석 틀
- 고객/현업 태클을 줄이기 위한 `UI Shell First` 구조 합의 자산
- MES 업무를 data-centric하게 재정의하는 canonical domain
- 공통 플랫폼 연계를 전제로 한 타깃 페이지 아키텍처 청사진, 행위 계약, 산출물 템플릿
- 공개 샘플을 통한 PoC 검증 동선

## 명확히 제외할 것

- 특정 고객사 UI 구조나 패키지 구조를 그대로 복제하는 일
- 샘플 프로젝트를 기준 정답처럼 삼는 일
- 반출 금지 자산을 저장소에 보관하는 일
- "한 번에 완전 변환"을 전제로 한 과도한 자동화 약속
- 공통 플랫폼이 담당하는 기능을 업무 페이지 저장소에서 중복 구현하는 일

## 성공 기준

이 저장소의 초기 성공 기준은 아래와 같다.

- 외부 지원 워크스페이스와 내부 단일모델 반입 번들의 역할이 혼동되지 않는다.
- 고객이 먼저 보는 `UI Shell`을 조기에 합의할 수 있다.
- 레거시 화면, dataset, transaction, controller, service, SQL을 같은 프레임에서 추적할 수 있다.
- 각 기능이 공통 플랫폼 제공 영역인지, 업무 페이지 구현 영역인지 구분할 수 있다.
- 고객사 고유 명칭 없이도 MES 공통 도메인으로 매핑할 수 있다.
- 타깃 산출물의 기본 형태를 "페이지 전환 단위"로 일관되게 생성할 수 있다.
- 구조 선고정과 행위 계약 잠금이 서로 다른 단계임을 설명할 수 있다.
- 샘플 검증 결과와 실제 현업 적용 절차가 분리되어 설명된다.

## 현재 샘플에 대한 해석

현재 `samples/ScoreRanking_Proj-master`는 다음 점 때문에 좋은 PoC 입력셋이다.

- MiPlatform XML 화면
- Spring Controller
- Service / DAO
- iBATIS SQL map
- dataset / transaction 호출 흔적

하지만 이 샘플은 어디까지나 "레거시 구조 분석이 가능한가"를 검증하는 용도다.
실제 본체는 여기서 추출한 기계적 패턴을 일반화한 프레임이어야 한다.
그리고 그 프레임은 공통 플랫폼 연계 규칙과 페이지 전환 규칙을 중심으로 재구성되어야 한다.
