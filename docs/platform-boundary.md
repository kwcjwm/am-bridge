# Platform Boundary

## 왜 이 문서가 먼저 필요한가

현재 전환 대상은 "새로운 MES 제품"이 아니라, 이미 존재하는 공통 플랫폼 위에서 동작하는 업무 페이지다.
따라서 AM 작업에서 가장 먼저 해야 할 일 중 하나는 공통 플랫폼이 책임지는 영역과 업무 페이지가 책임지는 영역을 분리하는 것이다.

## 공통 플랫폼 제공 영역

아래 항목은 기본적으로 공통 플랫폼이 제공한다고 본다.

- 메뉴
- 결재
- 권한
- 인증
- 메일
- 공통 레이아웃
- 공통 사용자 / 조직 / 코드 조회
- 공통 로그 / 감사 / 접속 정책

실제 프로젝트에서는 여기에 더해 공통 API 클라이언트, 공통 테이블, 공통 UI 컴포넌트가 있을 수 있다.

## 업무 페이지 구현 영역

업무 페이지 전환에서 직접 설계해야 할 범위는 아래다.

- 화면별 조회 / 저장 / 승인 / 처리 흐름
- dataset / transaction 기반 데이터 계약
- 페이지 상태 관리
- 화면별 입력 검증
- 업무 전용 API 연계
- 화면별 권한 체크 포인트
- 공통 플랫폼 기능 호출 방식

## 판단 원칙

### 1. 공통 기능은 재구현하지 않는다

레거시 소스에 메뉴, 로그인, 결재 팝업, 메일 발송 코드가 섞여 있어도, 타깃에서는 먼저 공통 플랫폼 제공 기능인지 확인한다.
공통 플랫폼 기능이면 업무 페이지 산출물에서는 "연계 포인트"만 남긴다.

### 2. 납품 단위는 페이지다

실제 컨버팅 실행 단위는 모듈 전체보다 페이지일 가능성이 높다.
따라서 inventory와 산출물도 페이지 단위로 끊어서 관리하는 편이 낫다.

### 3. 그래도 설계 기준은 데이터다

페이지 단위로 진행하더라도, 설계 기준은 여전히 dataset / transaction / service / domain이어야 한다.
페이지라는 단위는 납품과 추적의 기준이고, 구조화 기준은 데이터와 업무 흐름이다.

### 4. 백엔드는 선택적으로 본다

모든 페이지 전환이 신규 backend 개발을 의미하지는 않는다.
페이지에 따라 아래 중 하나일 수 있다.

- 기존 API 재사용
- 기존 backend 부분 개편
- 신규 API 추가
- 공통 플랫폼 API 직접 연계

## 표준 질문

각 페이지마다 아래 질문에 답할 수 있어야 한다.

- 이 페이지는 어떤 업무 객체를 다루는가
- 어떤 dataset과 transaction이 핵심인가
- 이 페이지가 사용하는 공통 플랫폼 기능은 무엇인가
- 그 기능은 호출만 하면 되는가, 확장이 필요한가
- 이 페이지에 신규 API가 필요한가
- 페이지 권한은 공통 권한 체계에 어떻게 연결되는가
