# Report Contract

## Canonical Rule

기본 보고서는 처음부터 한국어 메인 보고서 1본으로 작성한다.
이 보고서가 기본 정본이다.

CSV, 표, 부속 문서는 보조 자료다.
보조 자료가 메인 보고서를 대체하면 안 된다.

## Prompt Priority

보고서 항목 체계는 아래 우선순위를 따른다.

1. `prompts/amprompt.md`에 검증된 보고서 항목/형식 지시가 있으면 그것을 우선 사용
2. 없으면 이 문서의 기본 16개 섹션 구조를 사용

즉, `amprompt.md`는 선택이지만 존재할 경우 보고서 목차와 세부 항목 설계의 1순위 참조다.

## Main Report Rules

메인 보고서는 반드시:

1. 한국어로 작성한다
2. 한 파일로 읽을 수 있어야 한다
3. 상단에 목차가 있어야 한다
4. 각 주요 섹션에 설명 + inline 표가 있어야 한다
5. 상세 CSV나 부속 문서는 해당 섹션 안에서 링크해야 한다
6. `가서 찾아보세요`식 링크 허브가 되면 안 된다
7. shell/layout 판단과 behavior/API/SQL 판단을 분리해야 한다
8. 메인 시각 기준 이미지 파일명을 명시해야 한다

## Default 16-Section Outline

`amprompt.md`가 없으면 아래 구조를 기본으로 사용한다.

### 1. 문서 개요

- 페이지 ID / 페이지명
- 대상 XML
- 메인 이미지 파일
- 작성 기준일
- 작업 범위 한 줄 요약

### 2. 한눈 요약

- 이 페이지가 하는 일
- 이번 분석의 핵심 결론
- 가장 중요한 위험 3개 이내

### 3. 시각 기준 및 화면 복원 방침

- 메인 이미지 기준
- 추가 캡처 여부
- 이미지 누락 시 요청 여부
- shell 복원 기준

### 4. 화면 구조 요약

- 주요 영역 구분
- 상단/중단/하단 같은 고정 규칙이 아니라 실제 화면 기준 설명
- 주요 버튼/탭/팝업 진입점

### 5. 화면 구성요소 상세

- 주요 컴포넌트 표
- 그리드/폼/버튼/탭/팝업 진입 요소
- `5-4. 상세 목록`에서 전체 컴포넌트 CSV 링크

### 6. 사용자 동작 흐름

- 진입 시점
- 기본 조회 흐름
- 행 선택/상세 확인 흐름
- 저장/팝업/기타 액션 흐름

### 7. 데이터셋 요약

- 주요 데이터셋 설명
- primary/supporting 구분
- `7-4. 상세 목록`에서 datasets CSV 링크

### 8. 바인딩 및 화면-데이터 연결

- 어떤 화면 요소가 어떤 데이터와 연결되는지
- 주요 binding 표
- `8-3. 상세 목록`에서 binding CSV 링크

### 9. 트랜잭션 및 이벤트 흐름

- 주요 transaction
- 주요 이벤트/핸들러
- `9-4. 상세 목록`에서 transaction/event CSV 링크

### 10. 백엔드 / SQL / API 후보

- 백엔드 후보 흐름 요약
- SQL/mapper/service/controller 후보
- `10-4. 상세 목록`에서 backend candidate CSV 링크

### 11. 검증 / 상태 / 메시지 규칙

- validation
- enable/disable/state
- 예외/메시지
- `11-4. 상세 목록`에서 validation/message CSV 링크

### 12. 권한 / 팝업 / 연계 화면

- 권한 포인트
- 팝업
- 관련 화면 이동
- `12-4. 상세 목록`에서 navigation/popup CSV 링크

### 13. 전환 Shell / Mockup 방침

- 어떤 기준으로 껍데기를 그릴지
- placeholder 허용 범위
- 실제 구현 전까지 고정할 것 / 미룰 것

### 14. 전환 구현 판단

- FE/BE 분리
- 먼저 만들 것
- 나중에 잠글 것
- optional completeness support 사용 여부

### 15. 위험 / 미확정 / 추가 요청사항

- 불확실성
- PM에게 다시 받아야 하는 것
- 이미지 누락 / 소스 누락 / 해석 충돌

### 16. 부록 및 상세 참조

- 각 섹션의 상세 CSV / 부속 문서 링크 모음
- 예:
  - `5-4. 상세 목록: components.csv`
  - `7-4. 상세 목록: datasets.csv`
  - `9-4. 상세 목록: transactions.csv`
  - `10-4. 상세 목록: backend-candidates.csv`

## Section-Level Detail Rule

각 주요 섹션은 아래 형식을 기본으로 한다.

- 짧은 설명
- 핵심 inline 표
- 필요 시 ASCII 구조도
- 마지막 줄에 `상세 목록:` 또는 `상세 참조:` 링크

예:

- `7. 데이터셋 요약`
  - 설명
  - 주요 데이터셋 표
  - `7-4. 상세 목록: [datasets.csv](...)`

## Inline Detail Rule

메인 보고서에는 중요한 것만 보이게 적는다.
하지만 너무 적게 적으면 안 된다.

기본 규칙:

- 표는 보통 3~7행 이상 확보
- “대표 몇 개만” 적고 끝내지 말 것
- long-tail 전체 목록은 CSV로 넘기되, 메인 보고서 안에서 반드시 연결할 것

## Exhaustive Appendix Rule

AI가 쓴 보고서가 방향은 맞지만 빠진 항목이 많을 때만 optional completeness support를 쓴다.

좋은 사용 예:

- full dataset list
- full component list
- full transaction list
- backend candidate list

이때도 CSV나 부속 문서는 메인 보고서를 보완하는 역할만 한다.

## Visual Source Rule

메인 시각 기준은 `<page>.xml` -> `<page>.jpg` 를 기본 규칙으로 본다.

예:

- `FF02_DtCode.xml` -> `FF02_DtCode.jpg`

메인 이미지가 없으면:

- 먼저 PM/operator에게 요청한다
- 요청 사실을 보고서 `3. 시각 기준 및 화면 복원 방침`에 적는다

## ASCII Rule

화면 구조, 블록 구분, 주요 흐름은 필요할 경우 ASCII 도식으로 표현한다.

예:

```text
+--------------------------------------------------+
| 검색영역                                         |
+-----------------------+--------------------------+
| 메인 그리드           | 상세 폼 / 보조 정보      |
+-----------------------+--------------------------+
| 하단 버튼 영역                                   |
+--------------------------------------------------+
```
